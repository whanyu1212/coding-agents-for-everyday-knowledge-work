# github-management Skill

This is the OpenCode-global GitHub management skill used by the demo. It combines
general GitHub repository management with the label-driven triage workflow.

Source location in the local setup:

```text
~/.config/opencode/skills/github-management/SKILL.md
```

```yaml
---
name: github-management

Use whenever the user asks to triage GitHub issues, move issues to needs-info/ready-for-agent/ready-for-human/wontfix, label issues, review a PR, summarize a PR, inspect checks, manage a GitHub backlog, clean up labels, draft an issue or PR comment, find stale items, or prepare release notes.

## Operating Principles

- Use GitHub's `gh` CLI as the primary interface.
- Read before writing: inspect issues, PRs, labels, linked context, checks, and repo conventions.
- Prefer existing repo conventions over generic taxonomy.
- Mutations require explicit user intent.
- During issue triage, every GitHub comment or issue body generated by AI must start with:

```markdown
> *This was generated by AI during triage.*
```

## Canonical Triage Labels

Every triaged issue should carry exactly one category label and one state label.
If state labels conflict, stop and ask the maintainer before changing anything.

Category labels:

- `bug` — something is broken.
- `enhancement` — new feature or improvement.

State labels:

- `needs-triage` — maintainer needs to evaluate.
- `needs-info` — waiting on the reporter for more information.
- `ready-for-agent` — fully specified, ready for an autonomous agent.
- `ready-for-human` — needs human implementation because of judgment calls, external access, design decisions, or manual testing.
- `wontfix` — will not be actioned.

Normal transitions: unlabeled -> `needs-triage`; from there to `needs-info`,
`ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to
`needs-triage` once the reporter replies.

## Bootstrap Labels

```bash
gh label list --limit 100
gh label create bug             --color d73a4a --description "Something is broken" || true
gh label create enhancement     --color a2eeef --description "New feature or improvement" || true
gh label create needs-triage    --color fbca04 --description "Maintainer needs to evaluate" || true
gh label create needs-info      --color d4c5f9 --description "Waiting on reporter for more info" || true
gh label create ready-for-agent --color 0e8a16 --description "Ready for an autonomous agent" || true
gh label create ready-for-human --color 1d76db --description "Needs human implementation" || true
gh label create wontfix         --color ffffff --description "Will not be actioned" || true
```

Only run label creation when the user asked to set up or apply this workflow.

## Show What Needs Attention

Present three buckets, oldest first:

1. Unlabeled issues: `gh issue list --search "no:label" --state open`
2. `needs-triage`: `gh issue list --label needs-triage --state open`
3. `needs-info` with reporter activity since the last triage note: inspect comments for new reporter replies.

Show counts and a one-line summary per issue. Let the maintainer pick what to
work on.

## Triage A Specific Issue

1. Inspect the full issue with comments, labels, reporter, dates, and linked context: `gh issue view ISSUE_NUMBER --comments`.
2. Inspect repo labels, templates, CONTRIBUTING/SCOPE docs, and similar issues.
3. If `.out-of-scope/` exists, read `.out-of-scope/*.md` and surface any prior rejection that resembles the request.
4. For bugs, attempt reproduction before asking clarifying questions.
5. Recommend a category and state with evidence.
6. Wait for maintainer direction before applying labels, posting comments, closing, or creating files.
7. If underspecified, draft concrete, answerable questions and recommend `needs-info`.
8. If ready, write a durable brief before applying `ready-for-agent` or `ready-for-human`.

Outcome handling after explicit maintainer confirmation:

- `ready-for-agent` — post an agent brief comment, then apply labels.
- `ready-for-human` — post the same brief structure plus why it is not delegated, then apply labels.
- `needs-info` — post triage notes, then apply labels.
- `wontfix` bug — post a polite explanation, close as not planned, and label.
- `wontfix` enhancement — update `.out-of-scope/`, comment with linked reasoning, close as not planned, and label.
- `needs-triage` — apply the label; comment only if there is useful partial progress to preserve.

Use this for label transitions:

```bash
gh issue edit ISSUE_NUMBER --add-label LABEL --remove-label OLD_LABEL
```

## Needs-Info Template

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- specific, answerable question 1
- specific, answerable question 2
```

## Agent Brief Template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
Describe what happens now.

**Desired behavior:**
Describe what should happen after the work is complete.

**Key interfaces:**
- Interface, type, config shape, or behavior contract to inspect

**Acceptance criteria:**
- [ ] Specific, testable criterion

**Out of scope:**
- Adjacent thing that should not be changed
```

For `ready-for-human`, add `**Why not delegated:**` with the human-only reason.

Agent briefs should be durable and behavioral: avoid line numbers, brittle file
paths, and step-by-step implementation instructions.

## Reporting Shape

```markdown
**Category:** {bug | enhancement | unclear}
**Recommended state:** {needs-triage | needs-info | ready-for-agent | ready-for-human | wontfix}
**Recommended labels:** `{category}`, `{state}`, `{repo-native-labels}`
**Priority:** {P0 | P1 | P2 | P3 | unclear}
**Next action:** {what should happen next}
**Reasoning:** {short evidence-based explanation}
**Draft comment:** {only if useful}
```
